# Phase 0：安全与质量基线（第 1 周，P0）

## 执行目标

在不引入 Better Auth、用户体系、owner、review request 等大功能之前，先完成一轮**底座加固**，确保项目从“高风险可写”状态进入“可安全迭代”状态。

本阶段只解决以下问题：

1. 关闭匿名写入通道
2. 关闭数据库 public write RLS
3. 收口错误处理规范，修正明显语义问题
4. 建立迁移与回滚能力
5. 建立最小 smoke tests 回归护栏

诊断报告已确认当前普通 API 端点存在无认证写入风险，且数据库存在 `Allow public write access` 策略，因此这两项必须优先处理。 

---

## 本阶段不做

以下内容不属于 Phase 0，禁止提前展开：

* 不接入 Better Auth
* 不创建 `app_users`
* 不实现 owner / review_requests / audit_logs 新表
* 不新增登录页、用户管理页、路由守卫
* 不做大规模 controller / service 重构
* 不追求全项目 TypeScript strict 一次性完成

这些内容放到后续阶段执行。当前原计划把认证和所有权提前放在第一周，但诊断报告建议先做基础加固。 

---

## 输入依据

Agent 在执行本阶段时，以以下现状为事实基础：

* 原权限计划默认 Week 1 直接进入基础认证和路由改造。
* 诊断报告确认普通写接口存在无认证路径。
* 诊断报告确认数据库存在 `Allow public write access` 策略。
* 诊断报告建议优先补测试，尤其是 Word CRUD、同步逻辑、YAML 校验和 API 端点。
* 项目已有错误处理基础设施，不需要推倒重来。
* 项目已有 Pino 结构化日志和 request/user 上下文能力，错误处理规范应复用现有能力。 

---

## 执行原则

1. **先止血，再演进**
   先阻止匿名写入和 public write，再做后续认证升级。

2. **优先最小改动**
   本阶段尽量在现有架构上修补，不做大规模重写。

3. **允许过渡方案**
   在 Better Auth 接入前，允许继续使用受控 `ADMIN_TOKEN` 作为开发/管理写入口，但不得保留匿名写入。

4. **任何数据库修改必须可回滚**
   所有 schema / policy 调整必须附带回滚脚本。

5. **测试必须覆盖关键写路径**
   至少保证写接口认证和关键落库路径有自动化验证。

---

## 执行顺序

严格按以下顺序推进，不要跳步：

1. 审计写接口
2. 审计 RLS 策略
3. 收紧写接口认证
4. 关闭 public write RLS
5. 收口错误处理规范
6. 新建迁移与回滚脚本
7. 增加最小 smoke tests
8. 跑通本地与测试环境验证
9. 输出变更说明

## 任务 0.1：审计并收紧写接口认证

### 目标

让所有修改数据的接口都进入受控通道，不再允许匿名请求直接写库。

### 必做项

1. 审计所有写操作接口，包括但不限于：

   * `POST`
   * `PUT`
   * `PATCH`
   * `DELETE`
   * 任何具有副作用的特殊接口
   * 配置写入
   * 同步写入
   * 批量写入

2. 重点检查：

   * `routes/words.ts`
   * `routes/sync.ts`
   * `routes/core.ts`
   * 任何调用 `save/update/delete/execute` 的入口

3. 给所有写操作加统一受控鉴权

   * 在 Better Auth 上线前，使用现有管理鉴权或 `ADMIN_TOKEN` 作为过渡手段
   * 禁止匿名请求进入写逻辑
   * 未认证返回 401
   * 认证存在但无权限返回 403

4. 区分读写接口

   * 读接口暂不在本阶段强制上用户认证
   * 写接口必须全部经过鉴权

5. 检查隐藏写路径

   * service 内部是否暴露未受保护调用路径
   * sync 逻辑是否绕过 route 层直接写入
   * 配置端点是否已有受控鉴权且仍可正常使用

### 建议触达文件

* `web/routes/words.ts`
* `web/routes/sync.ts`
* `web/routes/core.ts`
* `web/middleware/*`
* `web/controllers/wordController.ts`

### 完成标准

* 任意匿名请求不能创建、更新、删除业务数据
* 所有写路由都能明确看到鉴权入口
* 现有管理接口仍可在受控凭证下使用

---

## 任务 0.2：关闭 public write RLS

### 目标

移除数据库层面对匿名/公共身份的写权限，避免数据库成为最后一个未受控写入口。

### 必做项

1. 审计当前所有业务表 RLS 策略，至少覆盖：

   * `words`
   * 未来会承接权限改造的主表
   * 配置、同步、日志相关表

2. 找出以下类型策略并处理：

   * `FOR ALL USING true WITH CHECK true`
   * 无条件允许 `INSERT / UPDATE / DELETE`
   * 对 `public` 或等效匿名身份开放写权限

3. 删除或替换 `Allow public write access` 一类策略。诊断报告已明确指出这类策略允许“任何人都可以写入”。

4. 制定过渡写入方案

   * 生产环境：不得存在 public write
   * 开发/测试环境：可以通过受控身份、迁移连接或管理凭证写入
   * 不允许以“为了开发方便”为理由保留匿名写策略

5. 验证关闭后行为

   * 未认证数据库写入失败
   * 合法受控写路径仍然成功
   * 同步、配置、脚本执行流程未被彻底阻断

### 输出要求

* 一份 RLS 审计清单
* 一份 policy 变更脚本
* 一份回滚脚本
* 一份过渡说明

### 完成标准

* 数据库中不再存在 public write RLS
* 匿名身份无法直接写入业务表
* 测试/开发流程在受控条件下仍可继续

## 任务 0.3：错误处理规范收口与边界修正

### 目标

在不推翻现有错误处理基础设施的前提下，统一错误边界和少数不一致行为，让后续认证和权限接入时具备可预测性。

### 背景判断

项目已经有以下基础设施：

* `AppError`
* `asyncHandler`
* 自定义错误类型工厂
* 全局错误处理中间件
* 404 处理

因此本任务不是从零设计错误处理，而是**规范收口**。

### 必做项

1. 明确 `asyncHandler` 使用边界

   * 统一约定只在一层包装异步错误
   * 推荐采用“controller 层统一包装，route 层仅挂载”的方式
   * 不做 route/controller 双重包装

2. 处理 `routes/words.ts`

   * 确认当前依赖 controller 内部 `asyncHandler` 的设计是否为项目标准
   * 如为标准，则保留
   * 通过注释、导出形式或命名让该约定更清晰
   * 不为了“表面统一”而重复包装

3. 处理 `routes/sync.ts`

   * 保留用于错误转换的局部 try-catch
   * 允许将底层错误转换成 `BadRequest` 等业务错误
   * 不允许吞错或返回模糊默认值

4. 处理 `routes/core.ts`

   * 修正数据库状态检查类接口的状态码语义
   * 依赖不可用时返回 503，而不是 200
   * 保留结构化日志记录

5. 统一关键错误响应结构

   * 通过全局错误处理中间件输出标准格式
   * 至少在关键接口上统一：

     * status code
     * error code
     * message

6. 补日志约定

   * 业务错误用 `warn`
   * 系统异常用 `error`
   * 日志中补充 request id、route、user id 等上下文
   * 不记录敏感数据内容。日志系统已支持敏感字段脱敏，应继续复用。

### 建议触达文件

* `web/routes/words.ts`
* `web/routes/sync.ts`
* `web/routes/core.ts`
* `web/controllers/wordController.ts`
* `web/utils/errors.ts`
* `web/middleware/errorHandler.ts`

### 完成标准

* 项目内 `asyncHandler` 责任边界明确
* 合理的局部 try-catch 被保留并规范化
* 数据库状态检查接口返回语义正确
* 关键接口错误结构稳定，可用于自动化断言

## 任务 0.4：建立迁移与回滚机制

### 目标

为后续 Better Auth 接入、业务用户表、owner 字段、review_requests 等 schema 变更提前建立可靠的迁移与回滚机制。

### 必做项

1. 明确 migration 目录与命名规范

   * 统一文件位置
   * 统一命名方式
   * 统一执行命令

2. 为本阶段数据库改动补齐脚本

   * RLS policy 变更脚本
   * 回滚脚本
   * 必要索引或安全修补脚本

3. 建立执行前检查流程

   * 备份数据库
   * 在测试环境先执行
   * 记录当前 schema 状态
   * 演练 up/down

4. 建立回滚约束

   * 禁止只写 up 不写 down
   * 禁止手工改线上库后不补 migration
   * 所有 policy 变更都必须进入版本管理

### 建议触达文件

* `schema.sql`
* `migrations/*`
* 部署或脚本目录中与 DB 初始化有关的文件

### 完成标准

* 可在测试环境完整跑通一次 up/down
* 本阶段所有数据库修改都有可执行脚本
* 后续 schema 演进不再依赖人工临时 SQL

## 任务 0.5：增加最小 smoke tests

### 目标

建立一组足够小但足够关键的自动化测试，用于验证本阶段的安全收紧没有破坏基本可用性。

### 测试重点

诊断报告明确建议优先覆盖 Word CRUD、同步逻辑、YAML 校验和 API 端点。
本阶段先不追求高覆盖率，只覆盖关键写路径。

### 必做项

1. 为以下路径补 smoke tests：

   * 匿名写接口被拒绝
   * 受控写接口可以成功
   * 非法 payload 返回正确错误码
   * 数据库关闭 public write 后，匿名落库失败
   * 合法写路径仍能完成创建/更新

2. 至少覆盖这些接口或行为：

   * `POST /api/words`
   * `PUT /api/words/:id`
   * 如已有删除接口，则覆盖 `DELETE /api/words/:id`
   * 同步路径中至少一个关键写流程
   * 一个数据库状态/健康检查接口

3. 测试分层

   * 单元测试：工具函数、局部 service 行为
   * 集成测试：关键 API 落库路径
   * 测试环境不得只依赖 mock，应至少有一组真实数据库行为验证

4. 使用现有测试框架

   * 诊断报告建议继续使用 vitest，并为 API 增加集成测试。

### 完成标准

* 至少有一组可重复运行的关键写路径 smoke tests
* 本阶段对鉴权、RLS、错误处理的改动都有测试保护
* 本地和 CI 可运行同一套最小测试命令

## 风险与处理策略

### 风险 1：过早关闭 public write，导致本地开发无法写库

处理方式：

* 先实现受控写入口，再移除匿名写权限
* 为开发/测试环境保留受控凭证或迁移身份
* 禁止直接保留 public write 作为“临时方案”

### 风险 2：错误处理范围扩太大，拖慢 Phase 0

处理方式：

* 只修关键路径
* 不做全项目风格重写
* 优先修边界不清和状态码语义错误

### 风险 3：测试只测 mock，不测真实数据库行为

处理方式：

* 至少保留一组真实落库测试
* RLS 调整必须有真实 DB 验证
* 不接受“本地看起来能跑”作为验收

### 风险 4：迁移脚本不可回滚

处理方式：

* 所有 migration 都写 down
* 所有 policy 改动进入版本管理
* 先在测试环境演练，再进入主分支

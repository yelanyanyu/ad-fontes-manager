# PRD: 2.0.9 Pipeline Workflow Boundary Optimization

## Problem Statement

Ad Fontes Manager 现在的 Pipeline 已经能支撑 Generate、Fix、Audit、Queue、Job History 和 Workset，但实现边界仍然偏散。Stage 的执行语义、Prompt 输入、Tool 解析、输出校验、修复和上下文写回已经逐步模块化，却还没有形成一个足够清晰、足够稳定的工作流接口。

这会带来两个问题。第一，后续要支持可自定义 Stage 时，开发者仍然容易回到“按 stage id 写分支”的旧路。第二，如果未来要把 LLM 调用、Tool loop、trace 或 repair loop 替换成 LangGraph、Mastra 或其他 agent workflow 框架，当前代码还没有一个足够窄的替换缝，迁移成本和回归风险都会偏高。

2.0.8 先暂停这项优化。2.0.9 的目标是把 Pipeline 收束成一个应用自有的小型 workflow runtime，让它保留现有用户体验，同时为未来自定义 Stage 和局部引入成熟 agent 框架留下清楚接口。

## Solution

2.0.9 会继续优化原有 Pipeline，而不是全量迁移到第三方 workflow 框架。

用户使用时不应该感知这次重构：Queue、Job History、Workset、partial YAML、Review Score、Fill Editor、Workset Save、Workset Improve 和 SSE 进度事件都应保持等价。开发侧则要看到更清楚的模块边界：Pipeline Definition 描述工作流，Stage Recipe 描述每个 Stage 怎么运行，LLM Stage Executor 负责通用执行，Tool Adapter 负责把 recipe 中声明的工具解析成运行时可调用能力，Runner 只负责顺序编排和最终结果组装。

这次优化不追求“一步做成通用开源 workflow 框架”。更实际的标准是：业务语义先从执行细节里解耦出来，让 Pipeline 可以继续被本项目使用，也可以在未来比较低成本地把内部 executor 替换为更成熟的 agent runtime。

## User Stories

1. As an Ad Fontes user, I want Generate to keep producing the same Word YAML, so that a technical refactor does not change my learning content.
2. As an Ad Fontes user, I want Fix and Improve to keep working from the same Queue and Workset controls, so that I do not need to learn a new workflow.
3. As an Ad Fontes user, I want partial Jobs to preserve usable partial YAML, so that a failed or stopped run can still be inspected and repaired.
4. As an Ad Fontes user, I want Review Score to keep appearing in Job History and Workset, so that I can continue deciding which results need attention.
5. As an Ad Fontes user, I want Fill Editor to behave the same after a completed Job, so that generated YAML still enters the repair surface safely.
6. As an Ad Fontes user, I want Workset Save to keep using the same save rules, so that generated content is not silently written with weaker validation.
7. As an Ad Fontes user, I want Workset Improve to keep selecting eligible low-score Jobs, so that improvement remains focused and predictable.
8. As an Ad Fontes user, I want Queue progress to keep streaming stage progress, tokens, tool calls, diagnostics, and completion state, so that long-running AI work remains visible.
9. As an Ad Fontes user, I want existing English and German pipelines to keep their current stage labels, so that the UI remains familiar.
10. As an Ad Fontes user, I want errors and stopped runs to remain recoverable from the same detail views, so that refactoring does not remove recovery paths.
11. As a maintainer, I want Stage behaviour to be declared by Stage Recipes, so that adding a new business flow does not require adding stage-id branches.
12. As a maintainer, I want the Runner to sequence stages without knowing prompt, tool, repair, or output parsing details, so that it stays small and easy to reason about.
13. As a maintainer, I want the LLM Stage Executor to own the common LLM-driven stage loop, so that searching, pondering, auditing, fixing, and future custom stages share one execution model.
14. As a maintainer, I want Tool Adapter to treat Tool as broader than function call, so that CLI commands, MCP tools, and custom callers can be added later.
15. As a maintainer, I want Tool Result to keep separate UI Result and Model Result channels, so that users can inspect richer evidence while the model receives compressed context.
16. As a maintainer, I want Pipeline Context Patch to be explicit, so that each Stage writes only the fields it owns.
17. As a maintainer, I want Prompt input assembly to be separate from stage execution, so that prompt variables can evolve without changing executor control flow.
18. As a maintainer, I want output validation and repair policy to be part of the Stage Recipe contract, so that different business stages can share one executor while using different validators.
19. As a maintainer, I want stop-loss behaviour to remain explicit, so that empty or degraded model output still fails in a controlled way.
20. As a maintainer, I want schema diagnostics to remain a content boundary concern rather than a Runner concern, so that editor and pipeline diagnostics stay consistent.
21. As a maintainer, I want stage progress events to be mapped through one adapter, so that future executor changes do not break the UI event stream.
22. As a maintainer, I want existing pipeline tests to move toward one high-level behaviour seam, so that refactors can happen without rewriting many implementation-sensitive tests.
23. As a maintainer, I want legacy pipeline definitions normalized before execution, so that old definitions keep working while new recipes become the stable contract.
24. As a maintainer, I want future single-stage full generation to be representable as a Stage Recipe, so that simple workflows do not need a special Runner.
25. As a maintainer, I want future phrase analysis to be representable as a different recipe over the same executor, so that reading words and constructing phrase analysis can reuse the same workflow shape.
26. As a maintainer, I want provider/model selection to remain behind provider boundaries, so that pipeline recipes do not leak provider configuration details.
27. As a maintainer, I want Queue to call Pipeline through a stable runner interface, so that Queue persistence and concurrency are not affected by executor internals.
28. As a maintainer, I want a future framework adapter to plug inside LLM Stage Executor, so that LangGraph, Mastra, or another framework can be evaluated without rewriting Queue and Workset.
29. As a maintainer, I want the app-owned Pipeline contract to remain the source of truth, so that third-party framework concepts do not leak into product behavior.
30. As a maintainer, I want this work deferred to 2.0.9, so that 2.0.8 can stabilize without another deep pipeline refactor.

## Implementation Decisions

- Keep the application-owned Pipeline instead of performing a full workflow framework migration in 2.0.9.
- Treat third-party agent or workflow frameworks as optional internal executor implementations, not as replacements for the app-visible Pipeline contract.
- Preserve the current user-visible behaviour for Queue, Job History, Workset, partial YAML, Review Score, Word save flows, and SSE progress.
- Make Stage Recipe the primary business-level stage contract. Stage kind should not become a growing enum of business differences.
- Keep the current stage labels for compatibility and UI continuity, but stop using stage ids as hidden execution semantics.
- Keep Pipeline Definition as the workflow description. It should declare stage order and stage recipes, not own low-level model/tool mechanics.
- Keep Runner responsible for sequencing, context merging, stopped/complete decisions, and final result assembly.
- Move LLM loop mechanics behind LLM Stage Executor. The executor should own model turns, tool loop coordination, output parsing, validation, repair attempts, and stop-loss handling.
- Keep prompt assembly as a separate concern from execution. Prompt variables and context enrichment should be testable without running a model loop.
- Treat Tool as a broad stage capability. Tool Adapter should be able to resolve function-call tools today and leave room for CLI, MCP, and custom tool callers later.
- Preserve the two-channel Tool Result idea: one result for UI inspection and one compact result for model context.
- Keep schema validation and diagnostics reusable across editor repair, manual repair dialogs, Workset save, and pipeline result handling.
- Introduce or strengthen public module entries so callers depend on stable contracts instead of internal files.
- Keep test-only injection behind explicit testing seams. Production module entries should stay narrow.
- Avoid new database schema changes unless a concrete user-visible behavior requires persistence. This PRD is primarily about module ownership and execution boundaries.
- Avoid building a visual workflow designer, custom user-authored stage editor, or plugin marketplace in 2.0.9.
- Keep comments in newly touched modules at file, function, and non-obvious implementation levels, using direct Chinese explanations rather than framework-heavy terms.

## Testing Decisions

- The preferred high-level seam is Pipeline Runner behavior with fake provider/tool adapters. Tests should assert external behavior: completed YAML, partial YAML, Review Score metadata, context patches, stopped outcomes, and progress events.
- Stage Recipe normalization should have focused tests proving that legacy stage definitions become explicit recipes without relying on stage-id inference during execution.
- LLM Stage Executor should have focused tests with fake adapters for model output, tool calls, repair attempts, stop-loss, output validation, and context patch generation.
- Tool Adapter should have focused tests proving known web tools resolve to runtime tools and unknown future tools remain representable without crashing the pipeline.
- Queue integration tests should remain at the Queue/Pipeline boundary: Queue creates Jobs, runs the runner, persists status, and emits expected state. They should not inspect executor internals.
- UI-facing behavior should be protected through existing Queue, Job History, Workset, and editor validation flows rather than snapshotting internal pipeline modules.
- Good tests should describe user-visible or boundary-visible behavior. They should not depend on private helper names, exact internal call order, or the file layout of pipeline modules.
- Tests that require SQLite native modules may need to run in the Electron-compatible Node ABI environment. Non-native module tests should remain runnable in ordinary development Node.
- Regression coverage should include the current English and German Generate flows, Fix flow, auditing score parsing, partial result preservation, and Workset Improve eligibility.

## Out of Scope

- No full migration to LangGraph, Mastra, Coze, or another workflow framework in this PRD.
- No visual workflow builder.
- No user-facing custom stage editor.
- No plugin marketplace.
- No new saved workflow format exposed to users.
- No database migration unless a later implementation slice proves it is necessary.
- No redesign of Queue, Job History, Workset, or YAML Editor UI.
- No change to Word Schema Version rules.
- No change to current English or German prompt content unless required to preserve behaviour during refactor.
- No attempt to make the Pipeline a standalone open-source framework in 2.0.9.

## Further Notes

This PRD follows ADR-0006, ADR-0007, and ADR-0008. The important direction is not “make everything generic”; it is “make the current business workflow explicit enough that future generic execution machinery can replace the inside without changing the outside.”

The 2.0.9 implementation should be split into small issues. A good order is: first lock down behaviour seams, then normalize recipes, then narrow public module entries, then deepen LLM Stage Executor, and only after that evaluate whether a framework adapter would actually reduce complexity.

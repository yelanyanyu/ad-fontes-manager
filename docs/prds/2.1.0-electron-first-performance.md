# PRD: 2.1.0 Electron-first Performance Optimization

## Problem Statement

Ad Fontes Manager 已经决定进入 Electron-first 方向：Electron 是唯一正式产品目标，Browser Debug Preview 只是开发调试入口。这个方向能减少 Web 产品兼容压力，但也带来一个新的责任：性能优化要按桌面应用来设计，而不是继续沿用“Web + 本地后端”的混合思路。

当前应用同时承担 YAML 编辑、Word Preview、AI Generate、Queue、Workset、SQLite 存储、更新检查和桌面打包。若启动阶段过早加载 CodeMirror、Preview、AI 配置、Queue 恢复、数据库和服务端模块，用户会感受到首屏慢、窗口出现后不可交互、AI 面板打开卡顿、Queue 大量历史恢复时 UI 不稳等问题。

这个迁移风险较大，不进入 2.0.8 或 2.0.9。2.1.0 才考虑正式推进。目标不是一次性重写 Electron 架构，而是先建立性能基线，再用小步改造让 Electron Product Target 变轻、变快、可测。

## Solution

2.1.0 会围绕 Electron Product Target 做性能优化。第一阶段先建立性能测量和验收标准，再优化启动路径、renderer 加载、后台任务、SQLite/Queue 边界和本地 API/IPC 边界。

用户视角的目标是：应用更快打开，更快进入可交互状态，打开 YAML Editor、AI Generate、Queue History、Workset 时更稳，不因为后台恢复、SQLite 查询或 AI Queue 事件而卡住界面。

工程视角的目标是：停止把 Web 当成正式发布目标后，优先优化 Electron 的真实路径。Browser Debug Preview 仍可存在，但不应驱动架构约束。任何高风险迁移，比如用 IPC 替代大量内部 HTTP API、把 SQLite/Queue 搬进 utilityProcess、或者彻底删除 Web build/start，都必须分阶段进行，并用性能基线和回归测试保护。

## User Stories

1. As an Ad Fontes desktop user, I want the app window to appear quickly, so that I do not feel the app is stuck during startup.
2. As an Ad Fontes desktop user, I want the first screen to become interactive quickly, so that I can begin reviewing or generating words without waiting for every background service.
3. As an Ad Fontes desktop user, I want AI Generate to open without visible stutter, so that starting a Job feels smooth.
4. As an Ad Fontes desktop user, I want YAML Editor to load only when I need to edit YAML, so that heavy editor code does not slow unrelated workflows.
5. As an Ad Fontes desktop user, I want Word Preview to render without freezing the UI, so that large generated entries remain inspectable.
6. As an Ad Fontes desktop user, I want Queue History to restore progressively, so that old Jobs do not block the whole app.
7. As an Ad Fontes desktop user, I want Workset to refresh predictably, so that today's review surface does not lag while background work runs.
8. As an Ad Fontes desktop user, I want update checks to happen quietly after the app is usable, so that startup is not delayed by network or release metadata.
9. As an Ad Fontes desktop user, I want settings and provider configuration to load when needed, so that provider setup does not slow normal reading or review.
10. As an Ad Fontes desktop user, I want long-running AI Jobs to keep updating progress without making the app unresponsive, so that I can keep working while generation runs.
11. As an Ad Fontes desktop user, I want SQLite-backed actions to feel local and fast, so that saving, loading, and filtering Words do not feel like remote web requests.
12. As an Ad Fontes desktop user, I want the app to recover after restart without a long blocking Queue scan, so that interrupted work does not punish the next launch.
13. As a maintainer, I want startup performance to have concrete numbers, so that optimizations are judged by measured improvement rather than intuition.
14. As a maintainer, I want Electron Product Target to be the performance baseline, so that Web-only behavior does not hide desktop regressions.
15. As a maintainer, I want Browser Debug Preview to stay useful for development, so that debugging remains convenient without keeping Web as a product.
16. As a maintainer, I want renderer chunks to load by workflow, so that CodeMirror, AI Generate, Settings, Anki export, and Preview do not all burden the first screen.
17. As a maintainer, I want main process work to stay minimal, so that window management and app lifecycle are not blocked by database or AI tasks.
18. As a maintainer, I want SQLite-heavy and Queue-heavy work to have a clear background boundary, so that later migration to utilityProcess can happen safely.
19. As a maintainer, I want Queue progress events to be throttled or summarized where appropriate, so that the renderer does not receive more reactive updates than it can display.
20. As a maintainer, I want internal HTTP and IPC boundaries to be evaluated by usage frequency and risk, so that the app does not perform a large transport rewrite too early.
21. As a maintainer, I want the existing save, import, Workset, and Queue behavior preserved, so that performance work does not change product semantics.
22. As a maintainer, I want native module ABI handling to remain reliable, so that performance work does not reintroduce Web/Electron ABI churn.
23. As a maintainer, I want CI to verify Electron-focused behavior, so that Web-only tests do not give false confidence.
24. As a maintainer, I want high-risk architecture changes split behind measurement, so that 2.1.0 can ship with controlled performance gains.
25. As a maintainer, I want performance documentation to explain what is measured and why, so that future changes do not drift back into startup bloat.

## Implementation Decisions

- Treat Electron Product Target as the only supported product baseline for performance work.
- Keep Browser Debug Preview as a development aid, not as a release target or architecture driver.
- Do not perform the migration in 2.0.8 or 2.0.9. This PRD targets a future 2.1.0 release cycle.
- Start with measurement before refactoring. The first slice should capture startup time, time to first window, time to first interaction, and major workflow open times.
- Optimize startup before deep architecture migration. Startup work should prioritize showing the window, rendering the first usable surface, and deferring non-essential initialization.
- Treat YAML Editor, Word Preview, AI Generate, Settings, and Anki export as candidates for route-level or workflow-level lazy loading.
- Treat Queue restore, Queue History load, Workset derivation, AI config loading, update checks, and prompt/schema reference loading as candidates for deferred background initialization.
- Keep main process responsibilities narrow: app lifecycle, BrowserWindow creation, preload bridge, menu/update coordination, and process orchestration.
- Evaluate SQLite and Queue background isolation after measurement. utilityProcess or another background boundary may be introduced later, but only after identifying actual blocking behavior.
- Preserve existing Queue, Job History, Workset, Word save, Word Import, and Preview Content semantics.
- Avoid a broad HTTP-to-IPC rewrite as the first step. If transport changes are needed, begin with high-frequency local calls where user-visible behavior is easy to test.
- Keep Express/internal server paths only where they still serve development, debugging, tests, or low-risk compatibility.
- Avoid changing the AI Pipeline contract in this PRD. Pipeline workflow optimization remains covered by the separate 2.0.9 Pipeline PRD.
- Keep native module ABI scripts available for Node-based tests and scripts even after Web is no longer a product target.
- Document every accepted performance budget and any deferred architecture migration, so future work has a stable baseline.

## Testing Decisions

- The preferred highest-level test seam is Electron user experience timing and behavior: app starts, first screen becomes usable, and major workflows open without breaking existing behavior.
- Add or script a repeatable local performance measurement path before making large changes. The measurement should record cold/warm startup where practical, first window time, first interaction time, and workflow open timings.
- Preserve existing renderer tests for editor diagnostics, AI Generate, Queue Table, Workset save results, Anki export, and Preview Content.
- Add focused tests only when a module boundary changes. For example, lazy loading should be tested through visible workflow behavior rather than private chunk names.
- If Queue restore or SQLite access moves behind a background boundary, test the public service boundary: request, response, cancellation, error propagation, and progress events.
- If internal HTTP calls are replaced by IPC for a workflow, test the workflow contract rather than the transport implementation.
- Include a desktop build smoke check before accepting performance work that touches Electron startup, preload, native modules, or packaging.
- Continue running type-check, lint, and the existing full renderer test suite after each implementation slice.
- Keep known Vue lifecycle warnings separate from performance acceptance unless they are proven to cause real UI jank.
- Do not accept performance changes based only on subjective feel. Each major slice should report before/after numbers or explain why measurement was not possible.

## Out of Scope

- No implementation work in this PRD.
- No GitHub issue creation yet.
- No 2.0.8 or 2.0.9 delivery commitment.
- No immediate migration to Electron Forge.
- No long-lived Web/Electron branch split.
- No full deletion of Web scripts in the first slice.
- No full HTTP-to-IPC rewrite in the first slice.
- No immediate move of SQLite or Queue into utilityProcess without measurement.
- No change to Word Schema Version rules.
- No change to AI Pipeline semantics.
- No UI redesign beyond changes needed to reduce load or defer heavy surfaces.
- No replacement of `better-sqlite3` as part of this PRD.

## Further Notes

This PRD follows ADR-0010: Electron is the supported product target, while Browser Debug Preview is a development aid. It also respects ADR-0009: pnpm and native ABI handling remain part of the build/runtime contract.

The safest 2.1.0 path is:

1. Measure first.
2. Make Electron startup usable sooner.
3. Lazy-load heavy renderer workflows.
4. Reduce noisy Queue/AI progress updates if measured as a renderer cost.
5. Only then evaluate utilityProcess or IPC migration.

The important discipline is to avoid turning “pure Electron” into a large rewrite. The win should come from removing Web product constraints and making the desktop path honest, measured, and incremental.

# Task Plan: Ad Fontes Manager 项目修复计划

## Goal
修复项目中的安全漏洞、明显 Bug，优化代码质量和性能，建立完善的开发规范和自动化流程。

---

## 项目背景

Ad Fontes Manager 是一个词源数据管理全栈应用，采用 Vue 3 + Express + PostgreSQL 技术栈，具有离线优先架构设计。经过全面诊断，发现 32 个问题需要处理，其中 7 个严重问题需要立即修复。

**核心原则**：
- **安全优先**：所有安全相关修复优先于功能优化
- **稳定可靠**：生产环境禁用调试功能，采用分层安全策略
- **渐进式改进**：不追求一次性重写，每个 Phase 独立可交付
- **向后兼容**：数据库 Schema 变更需要 migration 脚本

## Phase 1: 紧急安全修复（立即处理）

### 1.1 修复 `wordService.js` 变量使用错误

**问题描述**：
单词创建时，数据库 INSERT 语句使用了 `wordLower`（小写形式）而非 `yieldData.lemma`（原始形式），导致存储的单词丢失原始大小写格式。

**修复要点**：
- 修改 `addWord` 方法中的 INSERT 参数
- 使用 YAML 中定义的 lemma 值，如果不存在则回退到小写形式
- 保持查询逻辑不变（仍使用小写进行大小写不敏感匹配）

**影响范围**：
- 文件：`web/services/wordService.js`
- 功能：单词创建时的数据存储
- 风险等级：低

**验证方式**：
- 创建包含大小写的单词（如 "Apple"）
- 验证数据库中存储的是 "Apple" 而非 "apple"
- 验证查询功能仍然正常工作

### 1.2 为 `/config` 端点添加分层身份验证

**问题描述**：
配置更新端点 `/api/core/config` 可被任何人访问，允许修改数据库配置和本地存储设置，存在严重的未授权访问风险。

**安全策略**：
采用分层安全策略，区分生产环境和开发环境：

**生产环境**：
- 强制要求 `ADMIN_TOKEN` 环境变量已设置
- 验证请求头中的 `x-admin-token` 与配置值完全匹配
- 记录所有访问尝试（成功和失败）
- 未授权访问返回 401，未配置服务返回 503

**开发环境**：
- 允许无 Token 访问（方便调试）
- 或验证默认开发 Token
- 不强制要求环境变量

**环境变量要求**：
- 生产环境必须设置 `ADMIN_TOKEN`（建议 32 位以上随机字符串）
- 开发环境可选

**影响范围**：
- 文件：`web/routes/core.js`
- 功能：配置更新端点安全
- 风险等级：中（需要前端同步更新）

**验证方式**：
- 生产环境不带 Token 访问返回 401
- 生产环境带正确 Token 访问正常工作
- 开发环境无 Token 也能访问

---

### 1.3 限制 `x-db-url` 请求头功能 + 连接池安全

**问题描述**：
当前实现允许通过 `x-db-url` 请求头指定任意数据库 URL，存在 SSRF 攻击风险，且每次请求都创建新连接池，性能差且无连接限制。

**安全策略**：

**生产环境**：
- 完全禁止 `x-db-url` 请求头
- 强制使用环境变量 `DATABASE_URL`
- 记录所有尝试使用动态 URL 的请求

**开发环境**：
- 允许通过请求头指定数据库（用于测试）
- 创建临时连接池，不缓存

**连接池安全配置**：
- 最大连接数限制为 20
- 空闲连接 30 秒超时
- 连接建立 5 秒超时
- 防止连接池耗尽攻击

**影响范围**：
- 文件：`web/db/index.js`
- 功能：数据库连接池初始化
- 风险等级：低

**验证方式**：
- 生产环境设置 NODE_ENV=production
- 验证 x-db-url 请求头被忽略
- 验证使用配置的 DATABASE_URL
- 验证连接池限制生效

### 1.4 升级 Node.js 到 22 LTS + Express 5.0

**问题描述**：
- 当前 Node.js 18 已于 2025年10月 EOL，停止安全更新
- 原计划升级到 Node 20，但 20 将于 2026年4月进入维护模式
- Express 4.18.2 存在已知安全漏洞

**升级策略**：

**Node.js**：
- 升级到 Node 22 LTS（Active LTS 至 2027年4月）
- 在 package.json 中添加引擎限制
- 更新 Dockerfile 基础镜像

**Express**：
- 评估 Express 5.0 的破坏性变更
- 如果兼容性问题大，先升级到 4.21.x
- 逐步迁移到 5.0

**依赖检查**：
- 检查所有中间件的兼容性
- 验证 cors、body-parser 等常用中间件
- 运行完整测试套件

**影响范围**：
- 整个后端运行时环境
- 风险等级：中（可能有兼容性问题）

**验证方式**：
- 运行现有 API 测试
- 验证所有路由正常工作
- 检查是否有弃用警告
- 验证 Docker 镜像构建成功

### 1.5 添加 Helmet 安全头中间件

**问题描述**：
当前缺少 HTTP 安全头配置，存在 XSS、点击劫持等安全风险。

**安全策略**：
添加 Helmet 中间件配置：

**Content Security Policy (CSP)**：
- default-src: 'self'
- script-src: 'self', 'unsafe-inline'（Vue 需要）
- style-src: 'self', 'unsafe-inline'
- img-src: 'self', data:, blob:
- connect-src: 'self'
- font-src: 'self', Google Fonts CDN

**其他安全头**：
- HSTS（HTTP Strict Transport Security）
- X-Frame-Options（防止点击劫持）
- X-Content-Type-Options
- Referrer-Policy

**影响范围**：
- 文件：`web/server.js`
- 功能：所有 HTTP 响应
- 风险等级：低

**验证方式**：
- 使用浏览器开发者工具检查响应头
- 验证 CSP 没有阻止必要资源
- 运行安全扫描工具

**风险警告**：

- 所有团队成员需要协调同步
- 如果有其他敏感信息，一并清理

**影响范围**：

## Phase 3: 性能优化（本月完成）

### 3.1 添加数据库索引优化查询

**问题描述**：
LIKE 查询使用 `%` 前缀导致索引失效，数据量大时全表扫描性能差。

**优化方案**：

**索引策略**：
- 安装 pg_trgm 扩展
- 为 lemma 字段创建 GIN 索引
- 支持模糊搜索的索引优化

**查询优化**：
- 评估是否需要模糊搜索
- 如果不需要，改用精确匹配（B-tree 索引）
- 如果需要，使用 trigram 索引

**Migration 脚本**：
- 创建扩展（如果不存在）
- 创建索引
- 验证性能提升

**影响范围**：
- 数据库 Schema
- 查询性能
- 风险等级：中（需要测试）

---

### 3.2 配置 Axios 缓存机制

**问题描述**：
相同请求重复发送，浪费带宽和服务器资源。

**缓存策略**：
- GET 请求缓存 15 分钟
- POST/PUT/DELETE 不缓存
- 支持缓存失效

**实施方式**：
- 使用 axios-cache-adapter
- 配置合理的缓存时间
- 考虑缓存键的生成策略

**影响范围**：
- 前端请求模块
- 风险等级：低

---

### 3.3 优化 FontAwesome 导入

**问题描述**：
导入完整的 FontAwesome CSS（300KB+），但实际只使用少量图标。

**优化方案**：

**方案 A：按需导入 CSS**
- 只导入基础样式和使用的图标类型
- 减少首屏加载时间

**方案 B：使用 SVG 子集**
- 只导入实际使用的图标
- 更精细的控制

**推荐方案**：
先采用方案 A，如果体积仍然过大，再考虑方案 B。

**影响范围**：
- 前端构建体积
- 风险等级：低

---

### 3.4 添加请求防抖/节流

**问题描述**：
- 搜索输入频繁触发请求
- WordPreview 组件快速切换时重复加载

**防抖策略**：

**搜索输入**：
- 防抖 300ms
- 输入停止后才发送请求
- 取消之前的请求

**Word 切换**：
- 防抖 200ms
- 快速切换时只加载最后一个
- 使用 AbortController 取消请求

**实施方式**：
- 使用 lodash-es 的 debounce
- 或手写防抖函数

**影响范围**：
- WordList.vue
- WordPreview.vue
- 风险等级：低

### 3.6 优化计算属性和渲染性能

**问题描述**：
displayedRecords 计算属性每次全量计算和映射，数据量大时性能差。

**优化方案**：

**短期优化**：

- 减少 DOM 节点数量

**影响范围**：
- WordList.vue
- 风险等级：中

## Key Decisions

1. **Node.js 版本**：强制升级到 22 LTS，而非 20
   - 原因：Node 18 已 EOL，20 即将进入维护模式
   - 22 LTS 提供长期支持至 2027年4月

2. **安全策略**：采用分层策略
   - 生产环境：严格安全控制
   - 开发环境：便利性与安全平衡

3. **TypeScript 迁移**：渐进式而非一次性
   - 降低迁移风险
   - 允许逐步学习

4. **性能优化**：按需实施
   - 先解决明显的性能问题
   - 虚拟列表等高级优化根据数据量决定

5. **测试策略**：从单元测试开始
   - 优先覆盖工具函数
   - 逐步扩展到集成测试
